# 基准测试 (Benchmark) 说明

## 概述

基准测试脚本 `tests/benchmark.py` 用于衡量搜索引擎各组件的性能，并对比优化前后的算法效率。脚本分为三个部分，覆盖真实数据和大规模模拟数据。

**运行方式：**

```bash
python -m tests.benchmark
```

---

## 三部分测试结构

### Part 1：真实数据测试

使用 `data/index.json`（从 quotes.toscrape.com 爬取的 10 页、100 条名言、682 个词汇）。

**测试项目：**

| 测试类别 | 测试内容 | 示例查询 |
|---|---|---|
| 索引加载 | JSON 反序列化 + checksum 校验 | `storage.load()` |
| 单词搜索 | 单词查找 + TF-IDF 排名 + snippet 生成 | `find("love")` |
| 多词 AND 搜索 | 交集运算 + 排名 | `find("good friends")` |
| 精确短语搜索 | 位置索引验证词序 | `find('"the world"')` |
| OR / 排除 / 筛选 | 并集、差集、标签过滤 | `find("love OR fear")` |
| Trie 自动补全 | 前缀匹配 + 频率排序 | `suggest("lo")` |
| 拼写纠错 | 编辑距离计算 | `suggest("lovee")` |

### Part 2：大规模模拟数据测试

使用 mock 生成器创建 **500 页、5000 条名言、约 10000 个词汇** 的数据集，不需要网络连接。

**数据生成方式：**
- 词汇池 = 120 个常见英文词 + 10000 个随机生成词
- 每条名言从词汇池随机抽取 8-30 个词组成
- 作者从 20 个预设名人中随机选取
- 标签从 20 个预设标签中随机抽取 1-4 个
- 使用固定随机种子 (`seed=42`)，确保结果可复现

**目的：** 在真实数据太小（仅 10 页）的情况下，模拟更大规模的数据来暴露性能瓶颈，让优化前后的对比更有意义。

### Part 3：规模扩展对比

分别在 **50 / 200 / 500 页** 上运行相同查询，对比索引构建时间和搜索耗时如何随数据量增长。

## 算法优化对比

Benchmark 在同一数据集上分别运行优化版和未优化版算法，直接对比。

### 优化 1：AND 交集 — 小集合优先

| 版本 | 方法 | 策略 |
|---|---|---|
| 未优化 | `_intersect_terms_unoptimized()` | 按输入顺序依次取交集 |
| 优化后 | `_intersect_terms()` | 先按集合大小排序，从最小的开始交集 |

**原理：** 假设查询 `the love good`，其中 "the" 出现在 450 个页面，"good" 出现在 7 个页面。

- 未优化：先取 `the`(450) ∩ `love`(10) = 中间集合，再 ∩ `good`(7)
- 优化后：先取 `good`(7) ∩ `love`(10) = 很小的中间集合，再 ∩ `the`(450)

集合交集的时间复杂度是 O(min(|A|, |B|))，从小集合开始交集能让中间结果更小，后续交集更快。

### 优化 2：短语匹配 — `set` 替代 `list`

| 版本 | 方法 | 位置查找方式 |
|---|---|---|
| 未优化 | `_check_phrase_on_page_unoptimized()` | `positions` (list)，`in` 操作 O(n) |
| 优化后 | `_check_phrase_on_page()` | `position_set` (set)，`in` 操作 O(1) |

**原理：** 短语匹配需要验证"词 A 在位置 i，词 B 在位置 i+1"。查找某个位置是否存在于 posting 中：
- `list` 的 `in` 是线性扫描 O(n)，n = 该词在该页面出现的次数
- `set` 的 `in` 是哈希查找 O(1)

在 `PostingEntry` 中同时保存两者：
```python
@dataclass
class PostingEntry:
    positions: list[int]       # 保留有序列表，用于展示和遍历
    position_set: set[int]     # 新增哈希集合，用于 O(1) 位置查找
```

## 复杂度分析

| 操作 | 时间复杂度 | 说明 |
|---|---|---|
| 索引构建 | O(N) | N = 所有页面的总词数，遍历一次 |
| TF-IDF 计算 | O(W × P) | W = 词汇量，P = 平均每词的 posting 数 |
| 索引保存/加载 | O(S) | S = JSON 文件大小 |
| 单词查找 (print) | O(1) | dict 哈希查找 |
| AND 查询 | O(k × min(P_i)) | k = 查询词数，P_i = 各词的 posting 集合大小 |
| 短语匹配 | O(S × L) | S = 首词在该页面的位置数，L = 短语长度 |
| OR 查询 | O(sum(P_i)) | 取所有词的 posting 集合的并集 |
| Trie 自动补全 | O(K + M) | K = 前缀长度，M = 匹配结果数 |
| 拼写纠错 | O(V × L²) | V = 候选词数，L = 词长（编辑距离） |

## 测试结果示例

### 真实数据 (10 页, 682 词)

| 操作 | 耗时 |
|---|---|
| 索引加载 | ~9 ms |
| 单词搜索 | ~20-30 us |
| 多词 AND | ~15-25 us |
| 短语匹配 | ~13-33 us |
| OR 搜索 | ~30-45 us |
| Trie 补全 | ~2-45 us |
| 拼写纠错 | ~3500-4600 us |

### 大规模模拟 (500 页, 10092 词)

| 操作 | 耗时 |
|---|---|
| 索引构建 | ~290 ms |
| 单词搜索 | ~35-65 us |
| OR 搜索 | ~50-125 us |
| Trie 补全 | ~2-460 us |
| 拼写纠错 | ~33000-62000 us |

### 规模扩展对比

| 数据量 | 词汇量 | 构建时间 | AND 搜索 | 短语搜索 |
|---|---|---|---|---|
| 50 页 | 6,145 词 | 23 ms | ~9 us | ~11 us |
| 200 页 | 9,852 词 | 161 ms | ~9 us | ~9 us |
| 500 页 | 10,092 词 | 437 ms | ~12 us | ~14 us |

**关键发现：**
- **搜索查询时间几乎不随页面数增长** — 因为倒排索引使用 dict（哈希表），查找是 O(1)
- **索引构建时间线性增长** — O(N)，符合预期
- **拼写纠错从 ~3.5ms 增长到 ~62ms** — 词汇量从 682 → 10092，暴力编辑距离是 O(V)，可以通过 BK-Tree 优化但对当前规模不必要
